# 🌑 SHADOW-AI — 서울의 역설

> 서울은 혼자 살기 가장 편하고, 위험자를 가장 잘 찾는 도시다.
> 그런데 왜 5060 남성은 그 안에서 가장 많이, 가장 보이지 않게 죽는가?

서울 5060 남성 1인가구의 **보이지 않는 고립**을 두 축으로 **진단**하고, 행정동별 미래 위험을 **예측**하며, 지역 특성에 맞는 **처방**까지 제안하는 AI 기반 복지 의사결정 지원 프로젝트.

**SHADOW** = **S**eoul **H**idden **A**voidance & **D**ependency **O**verlap **W**atch

## 1. 문제 정의

| 단계 | 확정 사실 | 다음 질문 |
|---|---|---|
| ① 누구에게 | 2024년 전체 고독사의 **54.0%가 50·60대 남성**. 정의상 혼자 사는 사람의 죽음. | 어느 지역에서 두드러지나? |
| ② 왜 서울 | 인구당 발생률은 중상위권이나, **전체 사망 대비 고독사 비중은 전국 1위**. | 서울 5060 남성 1인가구의 생활 조건은? |
| ③ 취약 집중 | 단순노무 집중·이혼/사별·취약 점유 등 복합 취약성이 **동북·서남권 특정 자치구에 집중**(생계급여 밀도와 r=0.58). | 편리한 도시 인프라는 고립을 완화하나, 은폐하나? |
| ④ 제도의 공백 | 위험자 발굴은 많지만, 남성 고독사 사망자의 **62.2%가 복지 미수혜**. 닿은 경우도 77.6%가 공식 채널. | 가장 잘 찾는 도시에서, 정작 무엇이 닿지 못했나? |

→ EDA의 종착점은 **두 개의 충돌(역설)** 이다.

## 2. 두 개의 역설

> 서울은 혼자 살기 편하고 도움받을 구조도 있지만, **그 편리함과 제도성이 오히려 5060 남성의 고립을 보이지 않게 만든다.**

### 편의의 역설 (무의식적 고립)
편의 인프라(배달·간편식·생활편의)가 혼자 살아도 불편하지 않게 만들어 **외출·대면 접촉의 필요를 줄이고**, 고립을 본인도 사회도 모르게 고착시킨다.
> *가설:* "1인 편의 인프라가 잘 갖춰진 지역일수록 5060 남성 1인가구는 외출·소통이 줄고 고립이 고착된다."

### 복지의 역설 (의식적 고립)
복지 인프라와 발굴 체계가 있어도, **낙인·거부감·남성성 규범** 때문에 실제 이용으로 이어지지 않는다. 구조는 준비돼 있지만 당사자는 끝내 나타나지 않는다.
> *가설:* "복지가 손을 내밀수록 5060 남성은 오히려 더 깊이 숨는다."

### 회피의 생태계
```
[5060 남성의 도움회피·낙인회피] (퇴직·역할상실·자기은폐·남성성 규범)
        ┌──────────┴──────────┐
   선호: 익명적 접촉         회피: 식별 가능한 접촉
        ↓                      ↓
   편의 인프라 의존 ↑      복지 인프라 미이용
        ↓                      ↓
   사회적 접촉 강제력 소멸   보호 효과 미발생
        └──────────┬──────────┘
              [외출-커뮤니케이션 부재 = 보이지 않는 고립]
                          ↓
                  [고독사 위험 누적]
```

→ 그래서 고립 위험을 **하나의 점수로 합치지 않고 두 축으로 분리**한다.

## 4. SHADOW Map — 진단 (자치구·4분면)

두 축으로 서울 25개 자치구를 4개 고립 유형으로 분류한다.

### 두 축

**Dependency Index (편의 의존)** — 행정동 419개, 5요소 가중합 (100점)

| 가중치 | 요소 | 위험 방향 |
|---|---|---|
| 0.30 | 편의 인프라 밀도 (27개 업종 점포수, log) | 많을수록 ↑ |
| 0.25 | 외출·커뮤니케이션 모두 적은 비율 | 높을수록 ↑ |
| 0.15 | 배달·식재료 사용일수 | 많을수록 ↑ |
| 0.15 | 평일 이동횟수 (부호 반전) | 적을수록 ↑ |
| 0.15 | 독거 비율 | 높을수록 ↑ |

> 검증: 편의 인프라 ↔ 외출·소통 부재 Spearman r=0.643, 1인가구 비율 통제 후에도 부분상관 0.596 유지.

**Avoidance Index (복지 회피)** — 자치구 25개, 4요소 (서울서베이)

| 요소 | 의미 |
|---|---|
| 도움부재 | 도움받을 사람 없음 |
| 외로움부정 | 외로움·도움 필요성을 낮게 인정 (부정 갭) |
| 복지불신 | 제도에 대한 불신 |
| 네트워크축소 | 관계망 축소 |

> 핵심: 복지시설 밀도 ↔ Avoidance 무상관(rho=0.099). 회피는 **시설 부족이 아니라 낙인 문제**.

### Q1~Q4 유형과 처방 방향

| 유형 | 상태 | 처방 방향 |
|---|---|---|
| **Q1** 최고위험 | 의존↑ · 회피↑ | 낙인 낮춘 비대면 선제 개입 (에코-씰·우편 opt-out, 회신 불필요) |
| **Q2** 정서취약 | 의존↑ · 회피↓ | 50플러스·복지관·소셜다이닝 등 기존 사업 직접 연계 |
| **Q3** 상대안전 | 의존↓ · 회피↓ | 분기별 경량 모니터링 |
| **Q4** 자기해결 | 의존↓ · 회피↑ | 복지 언어 배제, 지역 상권·커뮤니티 기반 간접 발굴 |

> 진단(Q)은 자치구 단위로 결정된다 — Avoidance가 자치구 단위 데이터이므로, 행정동 Dependency를 자치구로 집계해 결합한다.

## 5. SHADOW AI — 예측 (행정동·전이)

SHADOW Map을 정적 진단에 그치지 않고, **현재는 안전하지만 곧 Q1으로 전이될 행정동**을 선제 예측하는 모델로 확장한다.

### 전이(Transition)의 정의
Q1이 아닌 행정동이 Dependency 궤적상 **Q1 기준선(전역 75번째 백분위)을 넘어서는 사건.** 두 관점으로 레이블링:
- `label_slope` — 4년간 Dep 상승세 상위 절반 (빠른 악화)
- `label_entry` — 시작은 안전했지만 최근 Q1 진입

### 피처 (시간 독립 ~13개)
| 그룹 | 예시 | 의미 |
|---|---|---|
| `level_*` | 외출커뮤·이동·배달·독거·인프라 (2022~2023 평균) | 행동 수준 |
| `delta_*` | 2022→2023 변화량 | 조기 악화 신호 |
| `infra_*` | slope·delta·recent | 인프라 장기 추세 |

> **데이터 누출 방지:** ① 레이블이 쓰는 `dep_*`·`dep_slope`는 피처에서 완전 제외 ② 레이블이 최신연도(2025) 기준이라 피처는 조기구간(2022~23)만 사용 ③ **전역 정규화**(4개 연도 풀링 후 1회 minmax)로 연도 간 점수 비교 일관성 확보.

### 모델 & 성능
Gradient Boosting (주) + Logistic Regression (기준선). N≈419로 작아 `max_depth=3·min_samples_leaf=10·subsample=0.8`로 과적합 억제. **Stratified 5-fold CV + 80/20 홀드아웃**(차이 0.05 미만 = 과적합 없음).

| 레이블 | 모델 | AUC | AP |
|---|---|---:|---:|
| label_entry | **Gradient Boosting** | **0.844** | 0.302 |
| label_entry | Logistic Regression | 0.842 | 0.271 |
| label_slope | Gradient Boosting | 0.759 | 0.628 |

### 위험등급 & 설명
| 등급 | 전이확률 | 의미 |
|---|---|---|
| 저위험 | 0.00~0.30 | 진입 가능성 낮음 |
| 중위험 | 0.30~0.50 | 모니터링 |
| 고위험 | 0.50~0.70 | 선제 개입 검토 |
| 최고위험 | 0.70~1.00 | 즉시 자원 배치 |

**SHAP**으로 행정동별 위험 동인을 분해(편의 의존 요인 기여도). 예측값은 **개인이 아닌 지역 단위 신호**.

> 보조: 행정동 모델은 Avoidance(자치구 데이터)를 못 담아, 자치구 규칙기반 보조점수(`0.5·dep_slope + 0.3·Avoidance + 0.2·Dependency`)로 보완.

## 6. SHADOW RAG — 처방 (뉴로-심볼릭)

진단·예측을 받아 **"이 지역에 어떤 제도적 복지가 필요한가"** 를 근거와 함께 제안하는 처방 모듈.

### 설계 철학: 한계 기반 이식 처방
기존 제도는 정답이 아니라 **한계를 가진 기준선**으로 본다.
```
① 현행 제도        "이런 제도가 (이미) 있습니다"
② 한계 진단 ⭐     "근데 낙인 유발 / 의존 심화 / 자치구 귀속 / 중복 배제로 부족합니다"
③ 필요 방향        "그래서 ~한 방향의 제도가 필요합니다"
④ 외부 레퍼런스    "그 방향은 해외/타지역/타자치구에서 이미 검증됐습니다"
⑤ 이식 제안        "이 지역에도 이런 제도가 마련되어야 합니다"
```

### 아키텍처: 온톨로지 지식그래프 + LLM (무환각)
- **검색(결정적):** 전문가가 설계한 복지 온톨로지(지식그래프)를 1~2홉 탐색해 **한계를 논리적으로 도출**. 두 축으로 모델링 —
  - 복지회피축: `제도 →자극→ 낙인요소 ∩ Q →민감→ 낙인요소 = 낙인 충돌`
  - 편의의존축: `제도 →심화→ 의존요소 ∩ Q →취약→ 의존요소 = 의존 충돌`
- **생성(LLM):** LLM이 검색된 근거 **안에서만** 처방을 작성 (제공 근거 밖 생성 금지 + 출력 스키마 강제 → 출처 필수).
- **벡터/임베딩 미사용** — 지식이 작고 구조적이라 그래프가 더 정확·설명가능. (해외사례가 대량으로 늘면 사례 검색에만 임베딩 추가 검토)

→ 한 줄: **"전문가 온톨로지로 한계를 결정적으로 진단하고, LLM이 외부 검증모델을 이식 처방으로 생성하는 무환각·설명가능 뉴로-심볼릭 시스템."**

> **현재 상태 (구현됨):** 처방 엔진 `shadow_rag_llm.py` 가 **OCI Autonomous Database(ADB)의 지식그래프 `shadow_graph`** 를 직접 쿼리해 사실을 결정적으로 추출하고, LLM이 그 근거로 처방문(①~⑤)을 생성한다. 담당자 질의응답 챗봇(`shadow_chat.py`)·그래프 시각화(`rag_graph.py`)도 대시보드에 연결됨. `kb/`(graph·prescriptions·cases)는 대시보드 데모용 그래프 모듈, `prescription/`은 본 구현 확장 자리.

## 7. 데이터 파이프라인 — shadow_profile.json

세 기둥의 산출을 **자치구 → 행정동 중첩 구조** 하나로 통합한 처방 인풋.

```jsonc
{
  "강남구": {
    "quadrant": "Q1", "dependency": ..., "avoidance": ...,   // 진단
    "회피분해": { "원본": {4요소}, "가중편차기여도": {...}, "절대가중기여도": {...} },  // 복지회피
    "의존분해": { "원본": {5요소} },                          // 편의의존 (Q 만든 그 구성요소)
    "행정동": [ { "행정동코드": ..., "dependency": ...,
                 "전이확률": ..., "위험등급": ..., "shap": [...] } ]  // 예측(SHAP)
  }
}
```

**분해 방법 (축마다 하나씩):** 복지회피 = 기여도 분해, 편의의존 = SHAP(전이 기여). 두 축의 **현재 구성(원본)** 은 모두 자치구 단위로 제공.

**빌드:**
```bash
python pipeline/build_gu_profile.py   # 인덱스 3종 → 자치구·행정동 골격
python pipeline/merge_profile.py      # + 전이예측·SHAP·기여도 병합 → shadow_profile.json
python pipeline/verify_profile.py     # 완전성 검증
```

### 클라우드 배포 (OCI) — 공개 시연

대시보드를 **OCI Compute 가상머신(VM)** 에 배포해, 내 노트북 없이도 누구나 접속할 수 있는 공개 웹서비스로 운영한다. (지식그래프는 OCI ADB, 앱은 OCI VM — 한 클라우드 안에서 연동)

```
[로컬]  코드+데이터+wallet  ──scp──▶  [OCI VM: Ubuntu]  streamlit run  ──▶  [OCI ADB: shadow_graph]
```

핵심 절차 (전체 가이드: **`deploy/README.md`**):
1. OCI Compute 인스턴스 생성 (Ubuntu, public subnet, 공인 IP)
2. OCI Security List에서 포트 `8501` 개방
3. `scp` 로 배포 묶음 전송 → `ssh` 접속
4. `bash deploy/vm_setup.sh` (파이썬·패키지·방화벽 자동 세팅)
5. 백그라운드 실행: `nohup streamlit run shadow_service.py --server.port 8501 --server.address 0.0.0.0 --server.headless true &`

> AWS EC2 배포와 동일한 IaaS 패턴. `wallet/`·`.env` 만 VM에 두면 ADB 처방 기능이 그대로 동작한다.
