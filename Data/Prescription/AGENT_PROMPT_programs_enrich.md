# 에이전트 수행 프롬프트 — programs.json 본문 구성 · 한계진단용 분류 · 검증 (end-to-end)

> 이 문서 전체를 Claude Code 에이전트에게 작업 지시로 넘긴다.
> 에이전트는 아래를 **그대로 작업 명세**로 삼아 자율 실행하고, 중단되면 끊긴 지점부터 재개한다.
> **단, 단순 필드 채우기가 아니다.** 이 데이터는 §0.5의 SHADOW 처방 흐름에서 **"현행 제도의 한계를 뽑아내는" 입력**이다. 한계가 보이는데 안 달면 처방 파이프라인이 헛돈다. 목적을 먼저 읽고 시작하라.

---

## 0. 역할 / 최종 목표

너는 데이터 큐레이션 에이전트다. 서울시 1인가구 참여프로그램 원자료(CSV 2개)와 이미 제도 단위로 묶인 골격(`programs.json`)을 받아, **각 제도의 `content`를 분석 방향에 맞게 재구성**하고 **분석 4필드(`access_mode`, `stimulates_stigma`, `deepens_dependency`, `fulfills_needs`)와 `rationale`·`status`·`target_age`를 분류·보정**한다. 전부 끝나면 **누락·중복·왜곡·오기입·과병합을 검증**하고 리포트를 남긴다.

**최종 목표는 "한계 진단이 가능한 데이터"를 만드는 것이다.** 각 제도가 우리 타깃(**서울 5060 남성 1인가구**)에게 어떤 **낙인 자극 / 의존 심화 / 연령·지역 불일치 / 중복배제**를 일으키는지가 드러나야 한다. 오래 걸려도 좋다. 단 **언제 끊겨도 작업물이 보존**되고 **끊긴 지점부터 바로 이어서** 실행 가능해야 한다.

## 0.5. ⭐ SHADOW 처방 흐름 (이 데이터가 쓰이는 맥락) — 반드시 먼저 이해

이 프로젝트(SHADOW-AI / SHADOW-MAP, 팀 Alonear)는 **서울 5060 남성 1인가구의 "보이지 않는 고립(고독사)"**을 다룬다. 두 역설 축:
- **편의의역설 (Dependency / 의존)** — 편의가 오히려 고립을 심화시킴 (행정동 단위 전이예측)
- **복지의역설 (Avoidance / 낙인·회피)** — 복지가 있어도 낙인 때문에 회피함 (자치구 단위 진단, Q1~Q4)

처방(SHADOW RAG)의 흐름은 **"한계 기반 이식 처방"**이다:

```
① 현재 제도         "이런 제도들이 (이미) 있긴 합니다"          ← programs.json (이 작업의 산출물)
        ↓
② 한계 진단 ⭐핵심    "근데 낙인을 유발 / 의존을 심화 / 특정 자치구 귀속 /
                      중복수혜 배제 / 연령이 안 맞음 → 이래서 부족합니다"
        ↓
③ 필요한 처방 방향    "그래서 ~~한 방향의 제도가 필요합니다" (needs)
        ↓
④ 외부 레퍼런스       "그 방향은 이미 여기서 검증됐습니다"
                      = 해외 / 서울 외 타지역 / 서울 내 타자치구(미시행)
        ↓
⑤ 이식 제안          "이 지역에도 이런 제도적 복지가 마련돼야 합니다"
```

**②에서 한계를 못 잡으면 ③④⑤가 굴러가지 않는다.** 그래서 네가 채우는 필드는 곧 ②의 입력이다. 아키텍처는 **뉴로-심볼릭**: `ontology.json`(지식그래프 규칙) + Oracle Graph(PGQL 충돌추론) + OCI Generative AI(생성). 벡터/임베딩 안 씀. 빌드타임(=지금 이 작업)에 LLM이 태깅하고, 런타임 그래프는 그 태그로 **결정적**으로 한계·이식후보를 매칭한다. 그래서 **태깅이 일관되고 근거가 명확해야** 그래프가 의미를 가진다.

### 네가 채우는 필드 → 어떤 ② 한계를 만드는가 (필드의 존재 이유)

| 필드 | → 만들어내는 ② 한계 | `ontology.json` 규칙 |
| --- | --- | --- |
| `stimulates_stigma` | **낙인충돌** = 제도가 자극하는 낙인 ∩ Q가 민감한 낙인 | `conflict_rules.stigma_conflict` |
| `deepens_dependency` | **의존충돌** = 제도가 심화하는 의존 ∩ Q가 취약한 의존 | `conflict_rules.dependency_conflict` |
| `regions` (특정 자치구) | **자치구귀속** (서울전체가 아님) | `structural_limits.자치구귀속` |
| `access_mode` = `중복배제` | **중복배제** (보편접근 막힘) | `structural_limits.중복배제` |
| `target_age` (중장년/전연령 미포함) | **연령 불일치 = 현행 아님(빈틈)** | `coverage_rule` |
| `fulfills_needs` | ③ 필요방향 충족 → ④ **이식후보 자격** | `transplant_rule` |

> ⚠️ **`ontology.json`이 코드·정의의 단일 원천(source of truth)이다.** 같은 폴더의 `ontology.json`을 작업 전 반드시 읽고, 코드 id·정의가 바뀌었으면 이 문서 부록 A보다 `ontology.json`을 우선한다.

## 2. 절대 규칙 — 잠긴 필드(수정 금지)

다음 두 필드는 결정론적으로 확정된 값이다. **읽어서 그대로 보존만 하고 절대 바꾸지 않는다.**

- `program_id` — 고유 ID. 변경·재부여·삭제 금지.
- `regions` — 시행 자치구(또는 `서울전체`). 추론으로 바꾸지 말 것. (자치구 섞임 버그 재발 방지)

> `source_urls`는 **스키마에서 제거됨**(그래프에서 안 쓰고, 죽은 폼링크/JS 포털이라 무의미). 출력에 넣지 않는다.

골격에 들어있는 `_meta`(담당기관, member_rows)는 원문 재결합과 검증에 쓰는 보조 정보다. **최종본에서는 `_meta`를 제거**한다(검증 통과 후).

## 3. 각 제도의 원문 전체 텍스트 확보 (content 재료)

골격의 `content`는 300자로 잘린 발췌다. 재구성하려면 **원문 전체**가 필요하다.

1. **CSV 재결합(유일한 본문 소스).** 골격의 제도는 `(담당 기관, 정규화제목)` 키로 묶은 것이다. 동일 규칙으로 CSV 두 파일을 다시 묶어 각 제도에 속한 모든 원행을 모은다. **`work/extract_source.py`가 이미 이 일을 한다** (START COUNT 인자로 구간 추출, 0건 미매칭 검증됨). 정규화 규칙:
   - 대괄호`[]`·소괄호`()`·HTML태그 제거
   - `숫자+(회차|회|기|차수|차|월|주|분기)` 제거, `20XX년?` 제거, 남은 숫자 제거, 공백 제거
   - 키 = `(담당 기관.strip(), 정규화제목)` — 빈 키면 원제목 사용
   매칭 행들의 `내용` 컬럼(HTML 제거 후) 전체 + 구조필드(`카테고리`·`모집대상 연령`·`모집대상 내용`·`진행방식`)가 content·분류 재료다.

   > ⚠️ **과병합 주의.** 정규화제목이 너무 공격적으로 줄어들면 **담당기관은 같지만 실제로는 서로 다른 프로그램**이 한 묶음에 섞일 수 있다(실측 사례: 강남 STAY.G의 `<일상이되는 정리>` 묶음에 `마음안심버스`·`여행작가`가 섞여 들어옴). content 작성 시 모은 원행들이 **같은 프로그램인지 확인**하고, 이질적이면 가장 대표적인(=골격 `name`과 가장 가까운) 원행 기준으로 작성한 뒤 `_audit`에 `"MERGE_SUSPECT": "이질 원행 N건 섞임"`을 남긴다.

2. **URL은 쓰지 않는다.** 원자료의 신청링크는 대부분 **죽은 모집 폼링크**(forms.gle 등)이고, 원천 포털(`1in.seoul.go.kr/.../sportView.do?sport_id=…`)은 **JS 렌더링이라 본문이 안 읽히고 비어서 온다**(검증 완료). 그래서 `source_urls` 필드 자체를 폐기했다. **URL 접속·크롤링 없이 CSV 본문만으로** 진행한다.

## 4. 필드별 작성 지침 (⭐ 적극 검수 — 한계를 적극적으로 드러내라)

> **태깅 철학(중요):** 이 서비스의 핵심은 **한계 발견(②)**이다. 그래서 "확실할 때만 보수적으로" 달지 말고, **제도의 설계 속성이 우리 타깃(5060 남성 1인가구)에게 장벽을 만들 가능성이 보이면 적극적으로 태그**한다. 단, **반드시 content의 근거가 있어야 하고**(추측·날조 금지 — 정직한 포지셔닝이 원칙), 모든 코드에 `_audit` 근거 1줄을 남긴다. 즉 **"근거 있는 적극", "근거 없는 확대해석 금지"** 둘 다 지킨다.
>
> 충돌은 **(제도 × Q) 쌍**에서 그래프가 결정적으로 계산한다(Q1/Q4만 특정 낙인에 민감, Q1/Q2만 특정 의존에 취약 — `ontology.json` 참조). 그러니 네가 폭넓게 달아도 **관련 없는 Q에는 자동으로 충돌이 안 잡힌다.** 과태깅을 두려워하지 말되 근거는 남겨라.
>
> ⭐⭐ **시작 전 반드시 `gold_examples.json`(태깅된 표준 13건)과 `ontology.json`의 `tagging_guide`(규칙 ①~⑦)를 정독하고 그 톤·강도·경계를 그대로 따른다.** 아래 4-x는 요약이고, **경계가 헷갈리면 `tagging_guide`가 단일 원천**이다. 특히: 대면→FaceExposure / 비대면→NoContact·NoOuting(rule1) · RelationRestore는 관계망 또는 생활자립역량만(rule2) · UniversalAccess·Transferable은 수동 태깅 안 함=그래프 자동파생(rule3) · 연령/지역 세그먼트는 보편(rule4) · RecipientIdentity는 특정 취약군 명시 시만(rule5).

### 4-1. `content` (필수) — 재구성
모집공고 문구를 그대로 두지 말고, **"무엇을 · 누구에게 · 어떻게(전달방식)"** 한두 문장으로 압축 재서술한다.
- 제거: 인사말, 신청기간/마감/선착순/모집인원, 문의처, 날짜·회차, 이모지, 깨진 특수문자(`?`·`☎` 등).
- 유지: 프로그램 본질(무엇), 대상(누구), **전달방식(대면/비대면/방문/온라인, 1:1/집단)** ← 분류 근거라 꼭 남김.
- **길이: 2~3문장(대략 100~200자).** 단순 요약이 아니라 **끝에 한계/성격을 박아 태그가 content에서 추적되게** 한다. 예: "…외출·접촉 없이 완결된다", "개인 단위라 관계 형성으로 이어지지 않는다", "청년 전용이라 중장년은 제외된다". (gold_examples.json의 content 톤을 그대로 따를 것)

### 4-2. `rationale` (선택) — 왜 효과적인가
content에서 드러나는 작동 원리를 한 문장으로. 근거가 약하면 빈 문자열 `""`. 추측 금지.

### 4-3. `target_age` (필수) — **CSV 근거로 적극 보정** (현행판정에 직결)
허용값: `청년`·`중장년`·`노인`·`전연령`. CSV `모집대상 연령` 버킷 매핑: `20대 이하|20~30대→청년`, `40~50대→중장년`, `60대 이상→노인`, 셋 다 포함→`전연령`.
- **골격값이 CSV·제목·대상과 어긋나면 CSV 근거로 보정한다**(골격이 mixed 원행에서 잘못 뽑힌 경우 많음). 예: 골격 `전연령`인데 제목이 `(중장년)`이고 대상이 `만40~67세`면 → `["중장년","노인"]`. 보정 시 `_audit.target_age`에 근거를 남긴다.
- **⭐ 이게 한계다:** `coverage_rule`상 **`중장년` 또는 `전연령`을 포함하면 우리 타깃에 "현행 인정"**, 아니면(청년·노인 전용 등) **현행 아님 = 빈틈**으로 잡힌다. 그러니 청년 전용 프로그램을 `청년`으로 정확히 좁히는 것 자체가 한계(연령 불일치) 발견이다. 데이터 왜곡이 아니라 **정확화**다.

### 4-4. `access_mode` (필수, 1개) — 분류
- `보편`: 대상이면 누구나(공개모집·무료·선착순·추첨, 자격심사 없음). 1인가구·거주 증빙, 멤버십(누구나 가입) 우선순위는 자격심사로 보지 않음 → 보편.
- `선별`: 자격심사·위기발굴·**순위/종합고려 선정**·특정 처지 한정(노부모 부양·가족돌봄·고독사위험 점수·소득심사 등)으로 **대상을 추림**.
- `중복배제`: 타 급여/사업 수혜자 제외, "타 관련사업 중복지원 불가", 기초수급 제외 등 **명시적 배제 조건**. → ② 중복배제 한계.
- 단서 없으면 `보편` + `_audit`에 `"access_mode: 단서없음→기본값"`.

### 4-5. `stimulates_stigma` — **적극** (낙인 = 복지의역설/회피 축)
값은 `ontology.json`/부록 A 코드에서만. content 근거 있는 것만, 단 **적극적으로**. 각 코드에 `_audit` 근거 필수.
- `STG_WelfareLabel`: 소득기준(중위소득·건강보험료)·기초수급/차상위·'취약계층'·'복지대상'·'위기' 라벨 노출, 또는 복지관 주최+취약 frame.
- `STG_FaceExposure`: **대면(오프라인 집합) 참여 필수** → 1인가구로 신상이 식별됨. (비대면/온라인 옵션이 주면 부여하지 않음) ← 5060 남성 핵심 회피요인이므로 적극 부여.
- `STG_SelfAction`: 본인이 직접 신청 + **서류·증빙 제출**(등본·소득·재직·자격 증빙). 단순 구글폼만이고 증빙 없으면 부여하지 않음.
- `STG_PublicExposure`: 공개 명단·동네 단위 공개행사·이웃에 참여가 드러나는 구조.
- `STG_RecipientIdentity`: **특정 취약 정체성으로 콕 집어 호명/선별** (고독사위험·돌봄부담·영케어러·노부모부양·위기가구 등). 단순 '서울 1인가구 누구나'는 제외(전 데이터 공통이라 무의미).

### 4-6. `deepens_dependency` — **적극** (의존 = 편의의역설 축)
- `DEP_NoContact`: 비대면 완결(온라인·전화·앱·IoT 안부)로 대면 접촉이 사라짐.
- `DEP_NoOuting`: 집에서 완결(비대면·배송·방문서비스)이라 외출 동기가 사라짐.
- `DEP_NoRelation`: **일회성/단발 수혜**(1회 행사, 물품·현금 지급, 일방적 제공)라 관계망이 안 생김.
- `DEP_LowEfficacy`: 제도가 다 대행/제공해 수동적 수혜 → 스스로 할 능력·의지 저하.
- `DEP_FragileSupply`: 상시 제공 전제(정기 배송·상시 돌봄)라 끊기면 바로 위기.

### 4-7. `fulfills_needs` — 강점(③ 필요방향 충족 = ④ 이식후보 자격)
한 제도가 한계와 강점을 **동시에** 가질 수 있다(예: 대면 소셜다이닝 = `STG_FaceExposure` 한계 + `NEED_RelationRestore` 강점). 이게 풍부한 진단이다.
**수동으로 다는 건 아래 2개뿐**이다(`tagging_guide.rule3`):
- `NEED_RelationRestore`: **(a)** 사람 간 관계망·공동활동·소통이 명시적(소셜다이닝·모임·함께 만들기) **OR (b)** 의존을 줄이는 **이전가능 생활자립역량**(디지털·재무·주거·생활기술 교육). ❌제외: 단순 체력·건강증진·개인취미·정서회복·자기돌봄·여가 → 빈 값. (`rule2`)
- `NEED_NoStigma`: 복지 라벨 없는 **생활·건강·취미·문화 언어 + 신상노출/대면 부담 낮음**(비대면·보편)일 때. 서울 현행엔 희소(=빈틈).

> ⚠️ **`NEED_UniversalAccess`·`NEED_Transferable`은 수동으로 적지 않는다.** 그래프가 `access_mode==보편` → UniversalAccess, `regions==서울전체` → Transferable 로 **자동 파생**한다(`rule3`). (서울 현행은 서울전체=0이라 Transferable은 항상 빈틈)

### 4-8. `status` (선택) — 보존
골격 값 유지(`운영중`·`종료`·`시범`). URL 접속을 안 하므로 적극 보정하지 않는다. 불확실하면 골격값 유지.

### 4-9. `evidence_ids` (선택)
연결 근거 ID가 실제로 있을 때만. 없으면 `[]`. 날조 금지.

## 4½. 본문 없는 제도 + 판단 로그 (일관성 핵심)

**본문이 비어있는 제도(약 173건):** 원본 CSV `내용`이 비었거나 광고 스크립트뿐인 경우다. **명칭·구분·모집대상·진행방식 컬럼으로 추론**해 분류하고, `_audit`에 `"content없음→명칭기반"`을 남긴다. content도 명칭·대상 기반으로 1~2문장 작성한다. 추론조차 불가능하면 해당 코드는 비우고 사유를 남긴다(억지 부여 금지).

**전달방식 불명 시(`tagging_guide.rule8`):** 본문/명칭에 `온라인·비대면·화상·ZOOM·앱·전화·우편·배송`이 **명시될 때만** 비대면(`DEP_NoContact·NoOuting`). 그 외에는 **대면으로 간주** → `STG_FaceExposure`. (서울 1인가구 센터 프로그램의 디폴트)

**⭐ 판단 로그(`tagging_guide.rule9`) — 반드시:** `tagging_guide`에 없는 **애매한 케이스**를 만나면,
1. 네가 정한 **판정 기준과 이유**를 `work/decisions_<담당>.md`에 1줄로 기록한다. 예: `P-0035 온라인 접수만 있고 강의방식 불명 → 대면 간주(rule8) / 취업특강 → 자립역량으로 RelationRestore`.
2. 그 기준을 **네 배치 전체에 일관 적용**한다(같은 유형을 다르게 찍지 말 것 = 플립플롭 금지).
3. 최종 제출 시 이 로그를 함께 낸다.

→ 4명 결과를 합칠 때 이 로그들을 비교해 기준을 통일하고 `tagging_guide`에 흡수한다. **애매할수록 "내가 어떻게 정했는지"를 남기는 게 정답** — 묻어두고 그냥 찍으면 4명이 따로 논다.

## 5. JSON 스키마 & 샘플 (반드시 일치)

각 제도 객체는 다음 키를 가진다(작성 중 보조용 `_audit`는 검증 후 별도 파일로 분리·제거):

```json
{
  "program_id": "P-0001",
  "name": "문자열",
  "content": "무엇을·누구에게·어떻게(전달방식) (재구성)",
  "rationale": "왜 효과적인가 (선택)",
  "regions": ["광진구"],
  "target_age": ["중장년", "노인"],
  "access_mode": "보편 | 선별 | 중복배제",
  "stimulates_stigma": [],
  "deepens_dependency": [],
  "fulfills_needs": [],
  "status": "운영중 | 종료 | 시범",
  "evidence_ids": [],
  "_audit": { "필드명": "근거 1줄 (검증 후 제거)" }
}
```

**스타일 기준 샘플(이 모양·톤·적극도에 맞춰라):**
- **의존충돌형(비대면)**: `"content":"1인가구에게 화상(ZOOM)으로 반찬요리를 함께 만드는 비대면 온라인 요리 프로그램."`, `"access_mode":"보편"`, `"deepens_dependency":["DEP_NoContact","DEP_NoOuting"]`, `"fulfills_needs":[]`.
- **낙인+관계복원 동시형(대면 소셜다이닝)**: `"content":"중장년 1인가구가 함께 요리·식사하며 관계망을 형성하는 소셜다이닝."`, `"access_mode":"보편"`, `"stimulates_stigma":["STG_FaceExposure"]`, `"fulfills_needs":["NEED_RelationRestore"]`. → 관계복원은 좋지만 대면강제라 회피형 5060엔 진입장벽(낙인충돌).
- **위기 호명+선별형**: `"content":"고독사 위험도가 높은 청년 1인가구의 고립은둔을 예방하는 활동 프로그램."`, `"access_mode":"선별"`, `"stimulates_stigma":["STG_RecipientIdentity"]`, `"fulfills_needs":["NEED_RelationRestore"]`.
- **중복배제형**: `"access_mode":"중복배제"`, `"stimulates_stigma":["STG_WelfareLabel","STG_SelfAction"]` (소득기준·증빙).
- **연령 빈틈형(이식 재료)**: `"target_age":["청년"]` → 중장년 미포함 = 현행 아님이지만 무충돌+`fulfills_needs`면 중장년용 이식후보로 살아남음.

## 6. 실행 파이프라인 (배치 + 원자적 저장 + 체크포인트)

핵심 불변식: **제도 1건 처리 = 파일 1개 원자적 기록 + 체크포인트 갱신.** 어느 순간 죽어도 잃는 건 처리 중이던 1건뿐이다.

1. 시작 시 `work/checkpoint.json`과 `work/enriched/` 목록을 읽어 **완료 ID 집합(done)**을 만든다.
2. `programs.json`을 `program_id` 오름차순 정렬, **done에 없는 것만** 처리 대상으로 큐잉.
3. `work/extract_source.py`로 해당 구간의 원문을 확보(또는 큐 단위로 호출). 배치 **15건** 단위로 돈다. 각 제도마다:
   a. §3로 원문 확보 → §4로 필드 작성(+`_audit`).
   b. `work/enriched/<program_id>.json`에 **원자적**으로 쓴다(임시파일 → `os.replace`).
   c. `checkpoint.json` 갱신(임시→replace): `{ "total":1743, "done":[...IDs], "failed":[{ "id":..,"reason":.. }], "updated_at":ISO }`.
   d. `run.log`에 `DONE P-XXXX` 또는 `FAIL P-XXXX <사유>` 추가.
4. 한 제도가 예외로 실패하면 `failed`에 기록하고 **다음 제도로 계속**(전체 중단 금지).
5. 모든 대상 완료 후 §8 검증.

> 체크포인트·결과 파일은 항상 `tmp 쓰기 → os.replace(tmp, 본파일)`로 교체해 부분기록 손상을 막는다.

## 7. 중단 감지 & 재개 프로토콜

**자기 보호(토큰/세션 한계 근접 시):** 신뢰 원천은 "자기 토큰 추정"이 아니라 **잦은 체크포인트**(제도 1건마다 flush)다. 강제 종료돼도 안전하다. 다만 컨텍스트가 매우 길어짐 / 요약·마무리 요청 / 도구 지연 누적 신호가 보이면 — **새 제도를 시작하지 말고** ① 진행 중이던 제도 flush, ② `run.log`에 `RESUME_AT <다음_미처리_program_id>`, ③ 재개 명령 출력 후 **깨끗이 중단**.

**재개:** ① `work/checkpoint.json` + `work/enriched/` 스캔으로 done 복원, ② done에 없는 가장 작은 `program_id`부터 §6 이어서 실행(멱등), ③ 로그에 `RESUME from P-XXXX (done=N/1743)`.

> ⚠️ **현실 견적:** 이 작업을 에이전트(사람-LLM)가 직접 한 건씩 신중히 분류하면 1,743건에 **여러 세션·다수 시간**이 걸린다. "하룻밤 무인 완료"는 보장되지 않으니, 반드시 체크포인트 멱등 재개로 나눠 진행한다.

## 8. 종료 후 검증 (누락·중복·왜곡·오기입·과병합)

`work/enriched/*.json`을 `programs.enriched.json`으로 병합(이때 `_audit`는 `programs.audit.json`으로 분리, `_meta`도 제거)한 뒤 모두 검사하고 `verification_report.{md,json}`에 남긴다.

**A. 누락 / 개수** — 출력 수 == 1,743. 모든 입력 `program_id`가 1:1 존재.
**B. 중복** — 중복 `program_id` 0건. 동일 `(name, regions)` 중복 의심쌍 경고.
**B-2. 과병합(신규)** — `_audit.MERGE_SUSPECT`가 있는 제도 목록, 그리고 `_meta.member_rows`가 비정상적으로 큰(예: 12+) 제도 경고(서로 다른 프로그램 병합 의심 — 사람 확인용).
**C. 보존 필드 무결성** — `program_id`·`regions`가 골격과 **완전 동일**. `regions` 값이 (서울 25개 자치구 ∪ `서울전체` ∪ 명시 국내·국가)에 속함. 출력에 `source_urls` 키가 남아있지 않은지 확인(폐기됨).
**D. 스키마 / enum 유효성**
- 필수 필드 존재: `program_id,name,content,regions,target_age,access_mode,stimulates_stigma,deepens_dependency,fulfills_needs`.
- `access_mode` ∈ {보편,선별,중복배제} (정확히 1개).
- `target_age` ⊆ {청년,중장년,노인,전연령}, 비어있지 않음.
- `stimulates_stigma`·`deepens_dependency`·`fulfills_needs` 각 값이 **`ontology.json` 코드 집합**에 속함(오타·미정의 코드 0건).
- `status` ∈ {운영중,종료,시범} 또는 생략.
**E. content 재구성 품질** — HTML·`&nbsp;`·이모지·깨진 특수문자(`?`·`☎`) 잔존 0건. 보일러플레이트(`안녕하세요|선착순|모집인원|신청기간|문의|☎`) 잔존 경고. 길이 이상치(0자/400자 초과) 목록. 원본 발췌와 80%+ 일치(재구성 안 됨) 경고.
**F. 분류 근거 추적** — 코드가 1개 이상인 제도는 `programs.audit.json`에 코드별 근거메모 존재. 누락 시 경고.
**G. 한계 커버리지(신규)** — 전체 중 ②한계가 하나도 안 잡히는 제도(`stimulates_stigma`=[] AND `deepens_dependency`=[] AND `access_mode`≠중복배제 AND `regions`=서울전체 AND `target_age`에 중장년·전연령 포함) 비율 리포트. **이 비율이 비정상적으로 높으면 적극 검수가 덜 된 신호** — 표본 재검토 권고.
**H. 요약 통계** — `access_mode` 분포, 코드별 부여 빈도, `target_age` 분포(현행/빈틈 비율), `status` 분포, 처리 시간.

위반(C·D) 1건이라도 있으면 상단 `❌ 검증 실패`, 없으면 `✅ 검증 통과(경고 N건)`.

## 부록 A. 코드 정의 (⚠️ `ontology.json`이 단일 원천 — 충돌 시 ontology 우선)

> 아래는 작업 편의용 사본이다. 실행 전 반드시 같은 폴더 `ontology.json`을 읽고, id·정의가 다르면 ontology를 따른다.

**stimulates_stigma — 자극하는 낙인** (Q1·Q4가 주로 민감 — `ontology.json: quadrant_sensitive_stigma`)
- `STG_WelfareLabel` 복지라벨: '복지' 라벨이 노출됨 (소득기준·취약·수급 등).
- `STG_FaceExposure` 대면노출: 본인 신상이 식별됨 (대면 참여 강제).
- `STG_SelfAction` 본인행동: 스스로 신청·방문·증빙해야 함.
- `STG_PublicExposure` 공개노출: 이웃에 알려질 위험.
- `STG_RecipientIdentity` 수혜정체성: '도움받는 사람'/특정 취약 정체성으로 규정·호명됨.

**deepens_dependency — 심화하는 의존** (Q1·Q2가 주로 취약 — `quadrant_vulnerable_dependency`)
- `DEP_NoContact` 비대면완결: 대면 없이 서비스가 끝남(정서적 고립).
- `DEP_NoOuting` 외출불요: 집 밖에 나갈 동기를 안 줌(물리적 고립).
- `DEP_NoRelation` 관계미형성: 일회성이라 관계망이 안 생김(사회적 고립).
- `DEP_LowEfficacy` 자기효능감저하: 제도가 다 해줘 스스로 할 능력·의지를 잃음.
- `DEP_FragileSupply` 중단취약: 상시 제공 전제라 끊기면 바로 위기.

**fulfills_needs — 채워주는 필요** (③→④, `limit_to_needs`/`transplant_rule`)
- `NEED_NoStigma` 무낙인: 낙인 자극 없는 전달방식(보편·생활언어).
- `NEED_RelationRestore` 관계복원: 관계망·자립을 회복시키는 설계.
- `NEED_Transferable` 이식가능: 특정 자치구에 묶이지 않고 이식 가능한 제도.
- `NEED_UniversalAccess` 보편접근: 중복수혜 배제 없이 보편 접근.

**access_mode** — `보편`(누구나) / `선별`(자격·위기·순위로 추림) / `중복배제`(타급여 배제 → 한계).
